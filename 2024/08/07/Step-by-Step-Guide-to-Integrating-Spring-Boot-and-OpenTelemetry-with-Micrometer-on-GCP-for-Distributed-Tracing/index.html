Trace Record Explanation

In the Jaeger trace record interface, you can see detailed information about a complete HTTP POST request to /api/orders. Here’s how to read and understand these trace records:

Overview

  • Trace ID: Each trace record has a unique trace ID that identifies the request.
  • Duration: The total time from the start to the end of the request. In this example, the duration is 2.53 seconds.
  • Service Count: The number of services involved in the request. In this example, there is 1 service involved.
  • Depth: Indicates the levels of sub-operations in the trace. In this example, the depth is 3 levels.
  • Total Span Count: The total number of spans in the trace. This indicates how many independent operation steps are involved in the entire request. In this example, there are 5 spans.

Services and Operations

Each trace record is broken down into multiple spans, with each span representing a specific operation. In this example, we can see the following main operations:

  • HTTP POST /api/orders: This is the starting point of the request, showing the total time to process this HTTP request (632.44ms).
  • create-order-request: The time taken to handle the order creation request (738.82ms).
  • create-order: The time taken to create the order (708.97ms).
  • get-order-status: The time taken to get the order status (201.51ms).
  • process-order-async: The time taken to process the order asynchronously (2 seconds).

Asynchronous Operation Records

In this trace record, we can see that asynchronous operations are also logged, which is very useful in Jaeger’s UI. The spans for asynchronous operations are displayed alongside synchronous operation spans, helping us understand what the application is doing when handling the request. For example, in the process-order-async span, we can see that this asynchronous operation took 2 seconds and overlapped with other operations.

Step 2: View the GCP Trace Interface

  1. Open GCP Console:
    Log in to your GCP account and open the Google Cloud Console.

  2. Navigate to Trace:
    In the left navigation bar, find and click on “Trace”.

  3. View Recent Trace Records:
    In the Trace interface, you can view recent trace records. Select your service name and set an appropriate time range to query the trace record of the request you just sent.

  4. Analyze Trace Data:
    In the Trace interface, click on a trace record to view detailed trace data. You should be able to see the start time, duration of each span, and detailed information about various services and operations. This data helps you analyze application performance and identify potential issues.

Trace Record Explanation

Selecting “Show expanded” in the GCP Trace interface allows you to see the integration of log messages and trace data. In this mode, each span not only displays its duration and related operations but also shows all log messages recorded within that span.

In this example, we can see several main operations:

  • HTTP POST /api/orders: This is the starting point of the entire request, showing the total time (2.578 seconds) for handling this HTTP request.
    • create-order-request: Time taken to handle the order creation request (806 milliseconds).
    • create-order: Time taken to create the order (752.585 milliseconds).
    • get-order-status: Time taken to get the order status (505.523 milliseconds).
    • process-order-async: Time taken for asynchronous order processing (2 seconds).

This information shows the detailed duration and related log messages for each operation, helping us better understand the application’s running status and perform performance optimization.

Summary

The integration of logs and traces allows us to have a more comprehensive understanding of the application’s running status and provides valuable information for optimizing the application’s performance and reliability.
